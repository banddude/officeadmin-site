Communications

Email Triage
Watches Gmail, classifies messages, drafts replies
▾

The comms pipeline scanner checks Gmail on a schedule. Morning scan at 7:45 AM, afternoon watchers at 3:00-3:45 PM. Each message is classified into one of three tiers.

Tier 1 messages are routine and predictable. These get auto-drafted and auto-sent at 9:30 AM without operator involvement. Tier 2 messages get a notification. Tier 3 messages need human review and get queued as Apple Reminders.

Scanner → Classifier → Dispatcher → Drafter (x5) → Approval Queue

Python Gmail API Claude Scheduled

Drafter
Writes in the operator's voice using learned rules
▾

The Drafter reads entity context, routing rules, and the operator's writing conventions before composing any message. It produces emails, iMessages, and other written communication that sounds like the operator wrote them.

There is a learning pipeline. When the operator corrects a draft, the correction is distilled into a rule and added to the drafting conventions file. Over time, the drafts get better without retraining anything.

Entity context + ROUTING.md + DRAFTING.md → Claude → Draft → Operator corrections → Updated rules

Claude Rule learning

iMessage
Two-way messaging with AIVA from a phone
▾

Reads the native macOS Messages database (chat.db) via SQLite for search and history. Sends messages through BlueBubbles, a cross-platform relay that acts as a channel provider.

The operator can message AIVA from their phone like texting a person. AIVA can also send proactive notifications. Access is controlled by an allowlist with pairing approval, so not just anyone can message the system.

Python SQLite BlueBubbles

Notify
Notification routing across all devices
▾

A Flask API server running as a daemon on the Mac Mini. It provides an HTTP interface for sending notifications via iMessage, Talk (voice), or agent sessions. Other modules call it when they need to get the operator's attention.

Python Flask Daemon

Talk / Canvas
Voice and visual output on connected devices
▾

Talk pushes text-to-speech notifications to all connected devices simultaneously using Apple Push Notification Service. AIVA can literally speak out loud through connected speakers.

Canvas is a phone-based HTML UI. AIVA can push interactive prompts, confirmation dialogs, and simple forms to the operator's phone screen. This is how AIVA shows information that does not fit in a text message.

Memory Architecture

AIVA has three distinct memory stores, a promoted-facts file, and scheduled dreaming passes that consolidate short-term recall into long-term knowledge.

1. Operational CRM
86 manually curated entities — source of truth for identity and operations
▾

A private GitHub repo serves as the working CRM. 86 structured entity folders, one per customer, vendor, crew member, or associate. Each entity has a JSON profile, tasks, and work records. Slugs are dashed: acme-corp, jane-doe.

Work lifecycle tracks from lead to payment: lead, bidding, bid, negotiating, awarded, scheduled, in_progress, complete, invoiced, paid. GitHub Actions run validators on every push. Email triage, the Drafter, daily sync, and contact URL linking all read from here.

This is the source of truth for identity and operational state. When a phone number comes in or a name appears in an email, it resolves to an entity here.

~/entities/ acme-corp/ entity.json # profile data tasks/ # task files work/ # job records, scope docs

GitHub JSON 86 entities Source of truth

2. OpenClaw Short-Term Memory
Recall cache with full-text search and vector embeddings
▾

OpenClaw maintains its own SQLite memory database at ~/.openclaw/memory/main.sqlite. Recent session content is chunked and indexed with both FTS5 (full-text search) and vector embeddings for fast semantic recall across recent work.

This is the fast-access layer. When AIVA needs to remember what happened in recent conversations, it queries this cache first. The data here is relatively recent and unfiltered — everything from the last few weeks of agent sessions.

This cache is also the primary input for the dreaming passes. Scheduled background jobs score and filter entries from here, promoting the most durable and frequently-referenced knowledge into long-term storage.

SQLite FTS5 Vector embeddings Semantic search

3. MemPalace Knowledge Graph
3,580 entities, 6,810 temporal triples — long-term historical memory
▾

MemPalace is an open-source SQLite knowledge graph. It holds the long-term historical memory of everything the system has processed: 3,580 entities and 6,810 triples with temporal bounds (valid_from/valid_to) so the system can reason about what was true when.

Entity types include person, company, tool, project, concept, place, property, event, and vehicle. All auto-extracted from Claude, Codex, and OpenClaw session logs by GLM-5.1 in chronological order. The .aiva/ layer wraps it with a thin module for ingest ordering, launchd scheduling, and priority queuing.

Entity IDs are underscored (acme_corp, quickbooks). The CRM is the working set for daily ops. MemPalace is the historical record of everything the system has ever known.

Open source SQLite GLM-5.1 3,580 entities 6,810 triples Temporal

MEMORY.md — Promoted Facts
Durable context loaded into every agent session
▾

A long-term promoted-facts file that sits at the root of the operational workspace. Dreaming passes write to it. Every agent session loads it as durable context, so AIVA always has access to the most important accumulated knowledge without needing to query a database.

This is the output of the memory consolidation pipeline: the distilled, high-confidence facts that survived the scoring and promotion process.

Markdown Auto-promoted Session context

Dreaming — Memory Consolidation

AIVA does not just respond to messages. It actively reviews its memory while the operator sleeps, promoting useful knowledge into durable storage — like how human sleep consolidates memory.

Memory Dreaming Promotion
3:00 AM daily — promotes short-term recalls into MEMORY.md
▾

The core dreaming job. Runs at 3 AM every day (currently enabled). Scores weighted short-term recalls from OpenClaw's memory cache and promotes entries that pass the threshold into MEMORY.md.

Promotion criteria: minimum score of 0.8, at least 3 recall counts, at least 3 unique queries referencing the entry, 14-day half-life decay, maximum 30-day window. Only the most durable, frequently-referenced knowledge makes it through.

OpenClaw memory → score + filter → promote → MEMORY.md

Cron 3 AM daily Enabled Scoring

Voice Pipeline
Transcription and speaker identification from recordings
▾

iPhone recordings sync via iCloud to the Mac Mini. A launchd watcher detects new files and triggers the transcription pipeline.

AssemblyAI handles speech-to-text with speaker diarization. Pinecone stores voice embeddings so the system can identify who is speaking across all recordings, even in new conversations. Transcripts feed into MemPalace for knowledge extraction and get linked to the relevant entities.

iPhone → iCloud sync → launchd watcher → AssemblyAI → Pinecone speaker ID → SQLite → MemPalace

Infrastructure

OpenClaw (open-source)
Agent orchestration, scheduling, message routing, MCP gateway
▾

OpenClaw is far more than a terminal UI. It is the agent orchestration layer that schedules every cron job, routes messages across channels (iMessage, Google Chat), hosts the MCP gateway that exposes AIVA's skills to agent sessions, and runs subagents in isolated workspaces.

Scheduler: 83 registered cron jobs (9 currently enabled) covering digests, email processing, habit tracking, dreaming passes, learning scans, and memory maintenance. Each job targets either the main agent session or an isolated subagent.

Message routing: Incoming messages from iMessage (BlueBubbles) and Google Chat arrive at OpenClaw's gateway, which dispatches them to the right agent session with full context. Channel-specific model overrides, typing indicators, and delivery policies are all configured per-channel.

MCP gateway: A local HTTP gateway (port 18789) that exposes all AIVA skills, tools, and modules as MCP endpoints. Agent sessions connect through this gateway to access the full system.

Short-term memory: OpenClaw maintains its own SQLite memory database with full-text search (FTS5) and vector embeddings. Recent session content is chunked and indexed for fast semantic recall. This cache feeds the dreaming passes that distill durable knowledge into long-term storage.

Subagents: Isolated agent sessions with their own configs and workspaces. Used for parallel task execution, background builds, and long-running research without blocking the main conversation.

~/.openclaw/ openclaw.json # gateway config, channels, hooks, skills cron/jobs.json # 83 scheduled jobs memory/main.sqlite # short-term recall (FTS + vector) subagents/ # isolated agent configs flows/registry.sqlite # workflow registry agents/main/sessions/ # session logs → MemPalace extract

Open source MCP Gateway Cron Message Router SQLite 83 jobs

Agent Runtime

▾

Background agent sessions run in tmux with per-agent workspaces and runtime configs. The agents CLI manages session lifecycle, the agent-revive CLI restarts crashed sessions, and the tell CLI sends commands to running sessions.

This is how AIVA handles tasks that take minutes or hours without blocking the main conversation. Long builds, research tasks, and multi-step workflows all run as background agents.
